РОЗДІЛ 0  —  Виправлення попередньої версії
Консолідація та виправлення архітектури
Цей документ є повною переробкою та консолідацією 39 архітектурних етапів попередньої версії. Усі виявлені суперечності, архітектурні розвилки та ризики вирішено та зафіксовано нижче.
Виправлені проблеми
1. Вирішена архітектурна розвилка: PostgreSQL vs Neo4j для CMDB
Попередня версія залишала невирішеним питання бази даних для топології мережі. РІШЕННЯ:
    • Фаза 1 (MVP): PostgreSQL з таблицею equipment_relations (relation_type: CONNECTED_TO, POWERED_BY, DEPENDS_ON). Достатньо для 2000 об'єктів та 20000 пристроїв.
    • Фаза 3 (після MVP): міграція топологічного графу в Neo4j, якщо кількість об'єктів перевищить 2000 або з'явиться потреба у складних графових запитах типу "знайди всі пристрої, що залежать від цього маршрутизатора".
    • PostgreSQL залишається основною БД для всіх інших CMDB-сутностей незалежно від фази.
2. Усунута суперечність щодо n8n
Попередня версія в деяких місцях покладала на n8n бізнес-логіку (SLA, FSM). РІШЕННЯ:
⚠ n8n виконує ВИКЛЮЧНО: нотифікації, ETL-операції, інтеграції з зовнішніми сервісами та планові задачі. Будь-яка бізнес-логіка (SLA, Warranty, Dispatch) реалізується всередині FSM Service.
3. Уточнено межу між ERPNext і FSM
Попередня версія непослідовно відносила Service Tickets то до ERPNext, то до FSM. РІШЕННЯ:

6. Вирішена стратегія AI при недоступності LLM-провайдера
Попередня версія не описувала fallback-стратегію. РІШЕННЯ:
    • LLM Abstraction Layer з інтерфейсом, що підтримує OpenAI, Anthropic, Google Gemini, локальний Ollama.
    • При недоступності провайдера: Circuit Breaker → черга запитів → сповіщення → деградація до RAG-only пошуку без генерації.
    • AI-залежні функції (генерація КП, кошторисів) деградують gracefully — система продовжує працювати без AI.
    • Оновлення RAG-індексів: щоночі incremental, щонеділі full. Документи старші 24 годин позначаються як потенційно застарілі.
7. Додана стратегія навантаження та тестування (НОВЕ)
Відсутність load testing strategy була виявлена під час аналізу. РІШЕННЯ:
    • Load testing: k6 або artillery для API Gateway та критичних ендпоінтів (tickets, visits, equipment).
    • Performance budgets: P95 latency < 200ms для CRUD, < 500ms для list endpoints, < 2s для AI chat.
    • CI/CD інтеграція: load tests запускаються на staging перед деплоєм в production при зміні API.
    • Baseline: фіксується при MVP, порівнюється при кожному релізі.

РОЗДІЛ 2  —  Фінальна архітектура системи
Архітектура системи
Принципи проектування
    • ERP Core залишається незміненим — жодних модифікацій ядра ERPNext
    • API First — будь-яка сутність доступна через REST API
    • Domain Isolation — FSM, CMDB, AI є окремими сервісами зі своїми БД
    • Zero Direct DB Access — сервіси не звертаються напряму до чужих баз даних
    • Event Driven — всі важливі зміни генерують події через NATS
    • Human in the Loop — AI пропонує, людина затверджує
    • Graceful Degradation — відмова AI не зупиняє ERP
Рівні системи
⚠ Всі клієнтські програми (Web, Android, Telegram) звертаються ВИКЛЮЧНО через Security API Gateway. Прямий доступ до сервісів заборонений.

Workflow Ticket
New → Triage → Assigned → Accepted → On Route → Working → Waiting Parts → Resolved → Closed
⚠ Таймер SLA стартує при створенні Ticket. Пауза дозволена ЛИШЕ у статусах Waiting Parts та Waiting Customer. При порушенні SLA: Telegram сервіс-менеджеру → email керівнику. Реалізується всередині FSM Service, НЕ в n8n.

Реалізація SLA Engine (НОВЕ)
SLA таймери реалізовуються всередині FSM Service через фоновий процес:
    • APScheduler: перевірка кожні 1-5 хвилин (конфігуровано)
    • Або pg_cron: PostgreSQL scheduler для перевірки SLA breach без зовнішнього процесу
    • Логіка: при кожній перевірці — порівняння поточного часу з SLA deadlines (sla_response_due, sla_resolution_due)
    • При breach: публікація події fsm.sla.breached в NATS → споживачі (Telegram, email) отримують сповіщення
    • Пауза SLA: при статусах Waiting Parts та Waiting Customer таймер зупиняється, накопичений час зберігається в sla_pause_minutes
    • При відновленні: deadline перераховується з урахуванням накопиченої паузи
    • Гарантія: якщо FSM Service недоступний — n8n Scheduler як fallback (але без business logic)

Принципи роботи AI (КРИТИЧНО)
⚠ AI НІКОЛИ не має прямого доступу до БД. Всі дані отримуються через Tool Calls → API Layer. AI успадковує права доступу користувача (RBAC). AI пропонує, людина затверджує.
    • Tool Calls: GetCustomer, GetTicket, GetObject, GetEquipment, SearchDocuments, SearchKnowledge
    • RAG: документи, договори, технічні паспорти, заявки, конфігурації
    • Fallback при недоступності LLM: деградація до RAG-only пошуку без генерації тексту
    • RAG оновлення: incremental щоночі, full rebuild щонеділі
    • Застарілі документи (>24 год без переіндексації) позначаються в метаданих

Вимоги безпеки
Автентифікація
    • JWT Access Token: термін 15 хвилин
    • JWT Refresh Token: термін 7 днів
    • MFA (TOTP): обов'язково для Owner, Director, Accountant; рекомендовано для всіх
    • SSH доступ до серверів: тільки SSH-ключі, пароль заборонений
    • Блокування акаунту: після 5 невдалих спроб на 15 хвилин
    • Автоматичний logout: через 8 годин неактивності
Шифрування
    • In Transit: HTTPS + TLS 1.3 для всіх з'єднань
    • At Rest: LUKS disk encryption на сервері
    • Файли MinIO: server-side encryption
    • Резервні копії: шифруються перед відправкою в offsite
    • Secrets: Docker Secrets або HashiCorp Vault (не зберігати у Git)
Аудит (логуються обов'язково)
    • Login / Logout / Failed Login
    • CRUD операції над критичними сутностями (договори, фінанси, права)
    • Експорт даних (CSV, Excel, PDF)
    • Зміни ролей та прав доступу
    • Всі API-виклики сервісних акаунтів (AI, Telegram, n8n)
Безпека файлів
    • Файли не публікуються напряму з MinIO
    • Доступ через Presigned URL з TTL 5 хвилин (фото) / 15 хвилин (документи)
    • Перевірка прав доступу виконується до генерації URL
Безпека Telegram
    • Прив'язка: кожен Telegram User → Employee (без прив'язки — доступ заборонений)
    • Першочергова верифікація через OTP при першому логіні
    • Інженер бачить через Telegram ЛИШЕ: свої заявки, виїзди, об'єкти свого регіону
    • Логується кожна команда
    • Обмеження Telegram API: файли до 50MB, медіа-обробка обмежена (для фото-звітів прийнятне)
Безпека AI
    • AI успадковує права поточного користувача (не суперюзер)
    • Захист від Prompt Injection: фільтр вхідних промптів
    • AI не може: закривати заявки, списувати матеріали, видаляти дані, змінювати фінанси
    • Всі AI-запити логуються (промпт, контекст, відповідь, вартість, модель)
    • Ліміти вартості: по користувачу, по відділу, по місяцю

РОЗДІЛ 6  —  API архітектура
API архітектура
Принципи
    • Єдина точка входу: Security API Gateway
    • Версіонування: /api/v1/ (підтримка версії мінімум 24 місяці)
    • Всі сервіси мають OpenAPI 3.1 специфікацію на /openapi.json
    • Стандартний формат відповіді: { success, data } або { success, error: {code, message} }
    • Pagination: limit + offset для всіх списків
    • Tracing: кожен запит отримує trace_id, що передається через всі сервіси
Security API Gateway (ДЕТАЛІЗОВАНО)
    • Стан: повністю stateless — всі дані сесії зберігаються в Redis
    • Circuit Breaker: для кожного бекенду окремий circuit breaker (tenacity або pybreaker)
    • Логіка: при 5 послідовних помилках сервісу → circuit open 30 секунд → half-open → закриття
    • Fallback: при недоступності FSM Service — повернення cached даних (якщо є) або 503 з Retry-After
    • Rate Limiting: Redis-based sliding window, ліміти в залежності від клієнта (див. Rate Limits)

LLM Development Rules
При використанні AI для генерації коду:
    1. Спочатку прочитати ERD та OpenAPI специфікацію
    2. Генерувати код ЛИШЕ в межах заданої архітектури
    3. Порядок генерації: DTO → Domain Model → Repository Interface → Service Layer → API Controller → Tests → Migration
    4. ЗАБОРОНЕНО: вигадувати нові сутності, змінювати ERD, порушувати Domain Boundaries
    5. При неясній специфікації: ЗУПИНИТИСЬ та запитати уточнення (не вгадувати)
    6. Кожна функція завершена тільки при виконанні всіх критеріїв DoD
